AWK — common pitfalls (detailed)
1
Initializing variables inside the body block
wrong
{
  max = 0        # resets every line!
  if ($2 > max) max = $2
}
END { print max }
correct
BEGIN { max = 0 }   # initialized once

NR > 1 {
  if ($2 > max) max = $2
}
END { print max }
The body block runs once per line. So max = 0 inside body resets max to 0 every single line before the comparison even happens. Final output will always be the last line's value, not the true maximum.

Rule: Running totals, max/min, counters meant to survive across lines — always initialize in BEGIN.
2
Printing NR in the body to get total line count
wrong
{
  print NR   # prints 1, 2, 3... per line
             # not the total!
}
correct
{
  # body — do your processing
}
END {
  print "Total lines:", NR
}
NR = Number of Records read so far. In the body, it's just a running counter (1 on line 1, 2 on line 2...). Only in END does it hold the final total, because all lines have been read by then.

Note: NR increments for every line regardless of whether your condition matched or not.
3
Using NR as denominator when file has a header
wrong
{ total += $4 }
END {
  avg = total / NR   # NR = 11, but 1 line
                     # was header, not data!
  print avg
}
correct
NR > 1 { total += $4; count++ }
END {
  avg = total / count  # only data lines
  print avg
}
If your file has 10 data rows + 1 header = 11 lines total. NR will be 11. Dividing by 11 gives wrong average. Always use your own counter that only increments for actual data rows.

Alternatively: total / (NR - 1) works only if you're sure there's exactly one header. Counter approach is safer.
4
Forgetting to skip the header — string becomes "max"
wrong — header gets compared
{ if ($2 > max) max = $2 }
END { print "Max age:", max }

# Output: Max age: Employee
# "Employee" > any number in AWK!
correct — skip header first
BEGIN { max = 0 }
NR > 1 {
  if ($2 > max) max = $2
}
END { print "Max age:", max }
AWK is very lenient with type comparisons. When comparing a string like "Employee" to a number, AWK doesn't throw an error — it just treats the string as greater. So your header row wins every comparison and becomes the "maximum".

Rule: Whenever doing numeric comparisons (max, min, sum, avg), always skip the header row with NR > 1.
5
Writing a variable inside quotes — prints literal text
wrong
END {
  print "Total salary sum"
  # Output: Total salary sum
  # (just the string, not the value!)
}
correct
END {
  print "Total salary:", sum
  # Output: Total salary: 450000
  # string label + actual variable
}
Anything inside double quotes in AWK is a string literal. The variable name is not interpreted inside quotes — it's printed as-is. Keep the variable outside the quotes, separated by a comma.

print "Label:", variable — this is the safe pattern to remember.
6
Using regex inside == (exact match operator)
wrong — regex not allowed here
$3 == /Engineer/  { print }
# This does NOT do regex match.
# == expects exact string only.

$4 == /[0-9]+/    { print }
# Also wrong — numeric range
# cannot be used with ==
correct
# For regex — use ~ (match) or !~ (not match)
$3 ~ /Engineer/   { print }
$3 ~ /^Eng/       { print }  # starts with Eng

# For exact string — use ==
$3 == "Engineer"  { print }

# For numeric comparison — use operators
$4 > 30000        { print }
== is for exact string or number equality. It needs the right-hand side to be a literal string in quotes.
~ is for regex matching — pattern goes inside /forward slashes/.
!~ means "does not match" the pattern.

Never mix them: $3 == /pattern/ won't work as expected.
7
Per-line counter initialized in BEGIN instead of body
wrong — for prime check per line
BEGIN { count = 0 }  # initialized once,
                      # never resets per line!
{
  n = $1
  for (i = 2; i < n; i++)
    if (n % i == 0) count++
  # count keeps growing across lines
  if (count == 0) print n, "prime"
}
correct — reset inside body per line
{
  n = $1
  count = 0   # reset for EACH new number
  for (i = 2; i < n; i++)
    if (n % i == 0) { count++; break }
  if (count == 0) print n, "prime"
  else            print n, "not prime"
}
This is the opposite of pitfall #1. For counters that should reset per line (like checking each number independently), initialize inside the body — not in BEGIN.

Rule of thumb:
— Cross-line accumulation (sum, max, total count) → BEGIN
— Per-line fresh start (divisibility count, flag) → inside body
8
sort — using lexicographic order for numbers
wrong — no -n flag
sort file.txt
# Lexicographic output:
# 100
# 23
# 45
# 9
# (100 comes first because '1' < '2')
correct — with -n flag
sort -n file.txt
# Numeric output:
# 9
# 23
# 45
# 100
By default, sort uses dictionary/lexicographic order — it compares characters left to right. So 100 comes before 9 because '1' < '9'. This is correct for words but completely wrong for numbers.

Always add -n when sorting numerical data. Add -r for descending order. Add -k2 to sort by the 2nd column.
9
sort with tab separator — missing the dollar sign
wrong
sort -k2 -n -r -t\t file.txt
# \t here is treated as TWO characters
# backslash + t, not a real tab.
# sort gets confused.
correct
sort -k2 -n -r -t$'\t' file.txt
#               ^ dollar sign needed!
# $'\t' tells bash to interpret \t
# as an actual tab character.

# For pipe separator — no dollar needed:
sort -k2 -n -r -t'|' file.txt
\t is a two-character sequence (backslash + letter t). The -t flag in sort expects a single character as separator. Without $, bash just passes the literal two characters, and sort doesn't recognize it as a tab.

The fix: $'\t' — the dollar sign triggers bash's ANSI-C quoting, which converts \t into an actual tab character before passing it to sort.

For single-character separators like |, ,, or space — no dollar needed.
